# capture-handover 拆解交接材料

把一坨交接材料（前手筆記、會議記錄、規格片段）拆成結構化條目，塞進對應領域包，讓 `learn-session` 之後能逐題研究。

## 開工前必讀
先讀 vault 根目錄的 `CLAUDE.md` 與 `筆記系統規範.md`（schema 與紅線唯一權威，可能已演進）。

## 定位 vault（依序，不得寫死絕對路徑）
1. Obsidian CLI：`obsidian vault list` / `obsidian eval vault="database" code='app.vault.adapter.basePath'`（用 Bash 時若找不到 node，先把 node 的安裝目錄加進 PATH）。
2. 查 memory / CLAUDE.md 記載路徑。
3. 都失敗才問使用者，並建議記進 memory。

## 紅線（違反即失敗）
- **不編造解釋**：材料裡沒講清楚的術語，白話解釋欄留空，交給 `learn-session` 研究。硬掰等於埋錯誤知識。
- **不代替理解、不總結成永久筆記**：只做拆解歸檔，永久筆記由使用者親手轉譯。
- 只寫術語表／問題佇列／決策日誌（`type: 專案筆記` 工作檔），不碰 `10_Permanent`、`30_MOC`。

## 流程
1. **確認目標領域**：看 `20_Project/` 下有幾個領域包。只有一個就直接用；多個就問使用者要歸到哪個；一個都沒有就提醒先跑 `learn-new-domain`。
2. **通讀材料，抽三類**（沿用各檔開頭 `<!-- 格式說明 -->` 的格式，別自創欄位）：
   - **專業術語**（不懂或首次出現）→ `術語表.md` 加列，狀態 `❓`。材料裡有解釋就摘錄到白話解釋欄，沒有就**留空**。來源欄註明出處。
   - **疑問與不確定處** → `問題佇列.md`。會卡住當下工作的放「🔥 阻塞中」，其餘放「📋 待研究」，每題附來源脈絡與日期。
   - **前手的隱性知識／警告／口傳決策**（「這個 flag 不能動」「燒錄前要先 X」）→ `決策日誌.md`，**忠實記錄不改寫**，附來源與影響範圍。
3. **去重**：加之前先掃現有內容，同一術語／問題別重複列。
4. **同步**：`git add` + `commit` + `push` 到 main，commit 訊息繁中簡述（例：`拆解 BIOS 交接會議記錄`）。
5. **回報**：新增了幾個術語、幾條問題（其中幾條阻塞）、幾則決策，以及問題佇列目前總量，並建議「接著用 `learn-session` 從 🔥 阻塞題開始研究」。

## 裁量
- 分不清是術語還是疑問時：純名詞放術語表，「為什麼／怎麼運作」放問題佇列，「不能動什麼／經驗」放決策日誌。
- 材料很長就分批處理，每批處理完回報一次，避免一次吞太多漏東西。
