# learn-session 每日學習引擎

領域包建好、材料拆解好之後，用這個 skill 每天啃 1-2 題：研究 → 白話講解 → 費曼驗證 → 歸檔。目標是「使用者真的懂」，不是「AI 幫忙寫完」。

## 開工前必讀
先讀 vault 根目錄的 `CLAUDE.md` 與 `筆記系統規範.md`（schema 與紅線唯一權威，可能已演進）。

## 定位 vault（依序，不得寫死絕對路徑）
1. Obsidian CLI：`obsidian vault list` / `obsidian eval vault="database" code='app.vault.adapter.basePath'`（用 Bash 時若找不到 node，先把 node 的安裝目錄加進 PATH）。
2. 查 memory / CLAUDE.md 記載路徑。
3. 都失敗才問使用者，並建議記進 memory。

## 紅線（違反即失敗）
- **不代寫永久筆記、不代寫轉譯內容**。學習產出只能以「靈感筆記」進 `00_Inbox`（標 🌱 待轉譯），由使用者親手轉譯進 `10_Permanent`——轉譯本身就是理解，外包給 AI 等於放棄理解。
- **`30_MOC` 與 `_學習地圖` 只建議、不代加**連結，等使用者點頭。
- 術語表「已懂」的白話解釋要用**使用者自己說出的用語**，不是 AI 的版本。

## 流程
1. **挑題**：讀 `問題佇列.md` 與 `術語表.md`，依優先序挑 1-2 題並向使用者確認（使用者可改選）：
   `🔥 阻塞中` > 在材料裡高頻出現的 `❓` 術語 > 最舊的「📋 待研究」題。為什麼確認：學習的主人是使用者，選題權在他。
2. **研究**：可用 WebSearch / WebFetch 查資料、讀使用者提供的文件。整理出**白話解釋**講給使用者聽——用類比、圖解、連結到他已知的概念，而不是照抄術語定義。
3. **費曼檢驗**：反問 2-3 個問題確認真懂，**應用題優於複述題**（「如果 X 壞了會怎樣」勝過「X 的定義是什麼」）。答錯就換個角度再解釋、再問。使用者說「跳過檢驗」可以跳，但提醒代價：沒驗證過的理解容易是錯覺，轉譯時會卡住。
4. **收尾**（沿用各檔 `<!-- 格式說明 -->` 的格式）：
   a. **問題佇列**：解決的題整行移到「✅ 已解決」、`[ ]`→`[x]`，下方縮排一行寫結論或指向 Inbox／永久筆記的 `[[wikilink]]`。
   b. **術語表**：狀態 `❓`→`✅`，白話解釋填**使用者通過檢驗時說出的話**。
   c. **值得留存的知識** → 寫一篇 `00_Inbox` 靈感筆記：`type: 靈感筆記`、`tags` 依詞彙表（含 `thinking/learning`，交接情境加 `handover`）、`source` 註明來源，正文標「🌱 待轉譯」並放本次問答的**原始材料**（你的講解 + 使用者的話），**明確告訴使用者：轉譯要自己動手**，別把它當成永久筆記。
   d. **建議連結**：口頭建議 `_學習地圖` 的知識骨架或某 MOC 可以掛哪些連結，但**不代加**。
5. **同步**：`git add` + `commit` + `push` 到 main，commit 訊息繁中簡述（例：`learn-session：搞懂 SPDM 握手流程`）。
6. **回報**：本次學懂什麼、更新了哪些檔、產出哪篇待轉譯靈感筆記，以及問題佇列剩餘量（阻塞幾題／待研究幾題）。

## 裁量
- 一次 1-2 題就好，貪多嚼不爛；使用者精神好且要求時才多啃。
- 研究查不到可靠資料時，誠實說「查不到定論」，把題留在佇列、別硬給答案。
