# scaffold-cards 轉譯鷹架

`learn-session` 產出的 `00_Inbox` 靈感筆記，要由使用者親手轉譯成 `10_Permanent` 永久筆記。這個 skill 只做**動筆前的準備**：確認拆幾張卡、查重、建好空檔、給連結候選，然後**退場**。

> 轉譯本身就是理解。這個 skill 存在的意義是把「開檔、抄 schema、想標題格式、翻舊筆記查重」這些雜事清掉，讓使用者的腦力全部花在真正的那一步——用自己的話重寫。

## 開工前必讀
先讀 vault 根目錄的 `CLAUDE.md` 與 `筆記系統規範.md`（schema 與紅線唯一權威，可能已演進）。卡片格式以 `90_Templates/永久筆記模板.md` 為準，**現場讀取那份檔案**，不要把格式寫死在本 skill 裡。

## 定位 vault（依序，不得寫死絕對路徑）
1. Obsidian CLI：`obsidian vault list` / `obsidian eval vault="database" code='app.vault.adapter.basePath'`（用 Bash 時若找不到 node，先把 node 的安裝目錄加進 PATH）。
2. 查 memory / CLAUDE.md 記載路徑。
3. 都失敗才問使用者，並建議記進 memory。

## 紅線（違反即失敗）
- **正文一字不代寫**。建出來的卡片，段落標題以下必須是空的。不摘要素材、不改寫素材、不「先擬個草稿讓你改」——擬草稿等於代寫，使用者改稿不是轉譯。
- **卡名由使用者取**。卡名就是那張卡的核心主張，取名＝想清楚，這步不能外包。AI 可以把素材裡**使用者自己寫的段落標題**列出來當參考，但不主動提議卡名、不建議該拆幾張。
- **連結只給候選、不代加**。`30_MOC` 與 `_學習地圖` 一律等使用者點頭。卡片內的連結候選寫在檔尾 HTML 註解區，由使用者取捨後自行刪除註解。
- **不刪原素材**。Inbox 原稿的去留由使用者決定（見流程 6）。
- 使用者中途要求「你先幫我寫一段」時：**拒絕並說明理由**（見 vault `10_Permanent/轉譯的意義.md`），改為提供反問、查重、格式健檢。

## 流程

### 1. 選素材
掃 `00_Inbox/`，列出標著 🌱 待轉譯的靈感筆記與各自放置天數（最舊優先）；`_轉譯工作台.md` 若當日有生成，以它的排序為準。請使用者選一篇——**一次只處理一篇**。

順手做**保鮮期失效自檢**（依 `靈感筆記的保鮮期與失效判定`）：放超過 3 天的，提醒使用者先自問「看不懂在寫什麼？」「看起來很無趣？」，任一為「是」→ 建議直接刪除，別硬轉譯。

### 2. 攤開素材
通讀選定的靈感筆記，在對話中輸出一張**素材段落清單**（只列使用者自己寫的標題／小節，不加你的詮釋），當作後續的核對底稿。

若素材含**來源可信度標註**（如 ✅實證／🟡推測 的分級表），特別提醒：**只有已實證的部分能寫進永久筆記**，推測性內容要嘛留在 Inbox、要嘛降級成待驗證問題丟回問題佇列。

### 3. 問使用者要拆幾張卡、各叫什麼
直接問，不代答。收到卡名後，就每個卡名向使用者確認一句「這張卡要收哪一段素材」，把對應關係記下來（供步驟 5 的核對清單用）。

若使用者把明顯多個主張塞進同一張卡，**提醒一次**「永久筆記傾向一卡一主張，這張看起來裝了 N 個」，但他堅持就照做——這是他的知識庫。

### 4. 查重
針對每個卡名，掃 `10_Permanent/`：
- 檔名層級：找語意相近的既有卡片。
- 內容層級：用 Grep 搜卡名關鍵字與素材裡的核心術語。

回報「這些主張可能已有卡片」，並讓使用者決定：**新開一張**、還是**補進既有卡**（後者就不建新檔，改為指出該檔位置讓他去補）。查重只報告，不替他合併。

### 5. 建空白卡片
對每個確定要新開的卡名，在 `10_Permanent/` 建檔：
- 格式套 `90_Templates/永久筆記模板.md`（現場讀取）。
- frontmatter：`created` 填今天、`type: 永久筆記`、`tags` 從素材繼承並限縮在詞彙表內（拿不準就留空讓使用者填）、`source` 指向素材靈感筆記或它的母材料 wikilink。
- **正文全空**，只留模板的段落標題。
- 檔尾加一段 HTML 註解區，放**連結候選**：

```markdown
<!-- 連結候選（Claude 建議，轉譯完自行取捨後刪掉本區）
- [[某永久筆記]] — 一句話說為什麼可能相關
- [[另一張卡]] — 一句話
-->
```

註解區不進正文、不影響 Obsidian 顯示，且明確要求使用者用完刪掉。

建完在對話中給一張**進度核對清單**（只在對話裡，不寫進 vault，避免汙染）：素材每一段 → 已認領給哪張卡 / 尚無人認領。

### 6. 交棒，然後閉嘴
明確告訴使用者：檔已備好，**接下來是你動筆**。可用 Obsidian CLI 開檔（`obsidian open`）。此時 skill 的主動工作結束，**不要繼續產出內容**。

待使用者說「寫完了」再進收尾，收尾做這些（他要哪個做哪個）：
- **費曼考官**：請他口頭講一遍這張卡在說什麼，反問挑洞（應用題優於複述題）。
- **格式健檢**：frontmatter schema、tags 是否在詞彙表內、wikilink 有無解析不到的。
- **回填**：領域包 `問題佇列.md` 對應題目那行，把 Inbox wikilink 補成／換成新永久筆記的 wikilink；`術語表.md` 來源欄同理。
- **原素材處置**：依規範，內容都收進卡片後**原靈感筆記應刪除**；提醒並執行使用者的決定，不自行刪。
- **連結建議**：口頭建議 `30_MOC` 或 `_學習地圖` 可掛的節點，等他點頭才動。

### 7. 同步
`git add` + `commit` + `push` 到 main，commit 訊息繁中簡述（例：`轉譯：SMM 保全室拆成 3 張永久筆記`）。

### 8. 回報
建了哪幾張卡、查重發現什麼、素材還有哪幾段沒認領、Inbox 剩幾篇待轉譯。

## 裁量
- 使用者只想開一張卡、不需要查重時，跳過步驟 4 直接建檔——別為了跑完流程而拖慢他。
- 素材很長時，步驟 2 的段落清單保持精簡（標題層級即可），細節讓他自己回頭看原檔。
- 若查重發現主張**已完全被既有卡片涵蓋**，誠實說「這篇可能不用轉譯，直接刪 Inbox 原稿即可」，別為了有產出而硬建空檔。
